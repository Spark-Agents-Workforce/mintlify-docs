**The anti-lesson this kills: "test everything all the time."** ([Offer testing](/growth-program/offer/offer-testing) established the doctrine; this lesson runs the whole program on it.)

Velocity culture says the team shipping the most tests wins. Subscription reality says the team shipping the most SIMULTANEOUS tests can't read any of them — because your instrument is the cohort table, cohorts take 60–180 days to speak, and two overlapping offer tests write on each other's cohorts permanently. Offer testing taught the one-dial rule for offers. The testing calendar extends it to the whole machine:

Slower than the growth-hacker fantasy. Faster than re-running every test twice because the first read was contaminated.

## The decision — sequence by leverage, read by maturity

(The sequencing and the surface rule are operator calls; the maturity mechanics import from the numbers and offer modules, the holdout from winback economics.)

**Why offers test first.** Biggest lever, slowest feedback: the offer decides who you acquire (the filter claim — measured at 50% vs 39% 180-day repeat between codes at Brand A, hedged as ever), and its verdict takes 180 days. Anything that slow and that heavy owns the front of the queue — start the clock early and let it run while faster tests cycle. **Then flows:** lifecycle changes read in weeks, and the compounding evidence (flows \$4.56M→\$7.57M YoY while broadcasts stayed flat) says flow improvements pay rent forever. **Then creative:** fastest feedback, shallowest permanence — creative fatigue means today's winner decays; test it continuously but never let it displace an offer or flow test from the calendar.

**The surface rule.** One test per surface — offer, PDP/store, lifecycle, paid — may run concurrently, because a PDP layout test and a welcome-flow test don't write on each other. Two tests on the SAME surface always queue. The calendar's job is saying no: the queue is where good ideas wait their turn instead of contaminating each other.

**Reading without fooling yourself.** The three disciplines, all built earlier, now enforced by calendar:

1. **Maturity windows:** 60-day = on-pace read only; 180-day = verdict. No offer test is declared won at week three. The calendar schedules the verdict date the day the test launches.
2. **Cohort contamination:** a mid-test discount event, a stacked holiday offer, a second dial moved — any of these re-writes the cohort you're reading. The calendar's contamination rule: tentpole windows (BFCM, the July stack) are marked NO-READ zones for offer tests; the data showed why — Brand A's holiday windows ran different terms entirely, and a cohort acquired mid-stack answers a question you didn't ask.
3. **The holdout:** for any intervention claiming to CAUSE retention (winback, save offers, perks), [the 10% randomized holdout](/growth-program/winback/winback-economics) is the only honest reader — attribution without one inflates everything it touches. The calendar allocates the holdout at launch, not after.

## The proof (and its honest edge)

The testing evidence in the data is mostly cautionary, which is exactly what a calendar needs: the measured filter effect that justifies offers-first (50% vs 39%), the flows-compound-broadcasts-plateau shape that justifies flows-second, the holiday-stack windows that justify no-read zones, and the no-holdout hedge on every attributed number that justifies the holdout import. No A/B infrastructure data exists in either store — the calendar is doctrine; the data supplies the physics it obeys.

## The artifact — the testing calendar template

```text The template
THE QUEUE (sequenced by leverage)
  OFFER LANE      one live test max · verdict at 180d · launch early, let it run
  STORE LANE      one live test max · read in weeks (PDP/cart/nav)
  LIFECYCLE LANE  one live test max · read in weeks (flows)
  CREATIVE LANE   continuous · never displaces the lanes above

AT LAUNCH (every test, no exceptions)
  [ ] Hypothesis + the ONE dial being moved (everything else frozen)
  [ ] Read date scheduled: on-pace check + verdict date on the calendar
  [ ] Holdout allocated if the test claims causation (the 10% design)
  [ ] Contamination scan: does the test window cross a tentpole? → shift or mark NO-READ

AT READ (the discipline)
  [ ] Read only at the scheduled maturity — no week-3 verdicts on 180-day questions
  [ ] Contamination check before conclusions: did anything else move mid-window?
  [ ] Decision written: ship / kill / extend — one line, dated (feeds the monthly)

THE STANDING RULE
  [ ] One test per surface. The queue is where ideas wait instead of colliding.
```
